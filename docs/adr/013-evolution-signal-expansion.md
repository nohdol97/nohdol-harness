# ADR 013 — 진화 트리거 4신호 체계 (수축·효율 신호 신설)

- **날짜**: 2026-07-14
- **변경 내용**: ① 진화 트리거를 3신호에서 **4신호**로 확장 — ④ 수축·효율(3주+ 무호출 스킬·에이전트·무참조 규칙 → 폐기·통합 제안, 산출 대비 토큰 과소모 패턴 2회+ → 절차 개선 제안, **주간 전체 점검 한정**). ② 반복 실패(②)의 정의에 **같은 내용의 사용자 정정 2회+** 포함. ③ 주간 무결성 점검에 **선언-미구현 스캔**(강제 장치 없는 규칙 목록화)과 **로딩 비대 점검**(SKILL.md 500줄 초과·항상 로드 문서 비대) 추가. ④ 신호는 4개로 고정 선언.
- **대상**: AGENTS.md 8절, harness-review(1·2단계·출력 형식·description), metaskill(진화 트리거·체크리스트), harness-review-reminder.py(일일 범위 문구), README
- **사유**: 사용자 질문(2026-07-14) "신호가 저 3개만으로 충분할까?" + "토큰 낭비도 줄이고 싶어" — 기존 3신호는 전부 확장(수요) 신호라 반대 방향(비대)과 옆 방향(구조 결함)을 못 본다.

## 결정

| 결정 | 내용 | 근거 |
|---|---|---|
| 수축 신호 신설 | 3주+ 무호출 스킬·에이전트·무참조 규칙 → 폐기·통합 제안 | 하네스가 매일 자라는데 안 쓰이는 것을 감지하는 신호가 없었다 — 미사용 정의는 우회조차 아니어서(쓸 일이 없었으니) 기존 3신호 어디에도 안 잡히고 로딩 토큰·유지 비용으로 쌓인다 |
| 효율 신호 동거 | 토큰 과소모 패턴 2회+(간단 작업에 팀 구성, 반복 수집, 과대 로딩) → 절차 개선 제안. 별도 5번째 신호가 아니라 ④에 통합 | 사용자 요청(토큰 낭비 감축). 신호 개수를 늘리면 점검 자체가 비대해진다 — 수축과 효율은 둘 다 "비용 대비" 관찰이라 한 신호로 묶인다 |
| 주간 한정 | ④는 일일 경량 모드에서 제외 | 3주 무호출 판정은 3주 관찰 창이 필요하고, stats 분석은 일일 "몇 분" 예산을 초과한다 — 일일 비용을 늘리지 않는 것 자체가 토큰 절약 목표와 정합 |
| 실측 전제 | ④의 무호출 판정은 agentsview 미설치 시 보류 | 근거 없는 폐기 제안은 근거 없는 신설 제안보다 위험하다(동작하는 것을 지운다) |
| 정정 반복은 ②의 확장 | 신호 신설 대신 반복 실패 정의에 "같은 정정 2회+" 포함 | 실패까지 안 간 마찰(같은 내용의 재지시)은 하네스가 의도를 놓친 신호인데 기존 정의의 사각지대 — 개수 증가 없이 커버 |
| 선언-미구현은 무결성으로 | 신호가 아니라 주간 무결성 점검 항목 | 티어 매핑 미적용·"병렬" 선언의 순차 실행·침묵 훅 — 이 프로젝트에서 반복 관찰된 최다 결함 유형인데 사용 데이터가 아니라 구조 검사 대상이다. 신호(수요 관찰)와 무결성(구조 검사)의 역할 분리 유지 |
| 4개 고정 | 8절에 "신호는 이 4개로 고정" 명시 | 신호 체계가 계속 자라면 점검이 비대해져 본말 전도 — 확장 3 + 수축·효율 1 + 구조(무결성)로 완결 |
