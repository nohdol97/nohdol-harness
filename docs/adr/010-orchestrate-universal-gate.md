# ADR 010 — orchestrate 범용 게이트화 (팀 필요성 판정 + 검증 필수 + 하이브리드 표준화)

- **날짜**: 2026-07-13
- **변경 내용**: ① orchestrate를 팀 작업 전용에서 **구현·다단계 작업 요청 전반의 관문**으로 확대 — Phase 0-1 팀 필요성 판정(직접 수행 / 단독 서브에이전트 / 생성-검증 쌍 / 팀 4등급)을 신설하고, 7절 라우팅 3항과 CLAUDE.md 트리거를 그에 맞게 개정. ② **검증 필수 규칙** 신설 — 기능 추가·동작 변경이 포함된 작업은 판정 등급과 무관하게 reviewer 독립 검증 Phase를 포함. ③ 하이브리드(C) 모드에 **표준 스켈레톤 5-Phase**(수집→설계·합의→구현→독립 검증→통합)와 patterns.md 하이브리드 구성 규칙 5가지를 신설.
- **대상**: AGENTS.md 7절, orchestrate SKILL.md(Phase 0 재구성·검증 필수 규칙·C모드), metaskill SKILL.md·references/patterns.md(복합 패턴 절 확장), CLAUDE.md(트리거·스킬 우선순위), README, harness-review SKILL.md(우회 신호 확장), team-review SKILL.md(임계 참조), doc-writer·branch-workflow SKILL.md(PR 템플릿), .agents/agents/implementer.md(검증 짝 필수)
- **사유**: 사용자 요청(2026-07-13) — "서브에이전트 하나만 돌려도 되는 작업 지시에도 orchestrate가 불려서 팀 필요 여부가 체크되면 좋겠다. 검토를 강화하고 패턴을 하이브리드하게 잘 구성할 수 있게." 기존 구조는 "팀이 필요한 작업만 orchestrate를 부른다"였는데, 팀 필요 여부를 판정할 주체가 없어 복잡한 작업이 판정 없이 단일 컨텍스트로 처리되는 우회가 구조적으로 발생했다.

## 결정

| 결정 | 내용 | 근거 |
|---|---|---|
| 관문화 | 구현·다단계 요청은 전부 orchestrate Phase 0-1을 거친다. 판정 없는 단일 컨텍스트 처리는 하네스 우회 신호(8절) | 판정은 판정 주체가 있어야 일어난다. "필요할 때만 부른다"는 필요 판단 자체를 누락시킨다 |
| 4등급 판정 | 직접 수행 / 단독 서브에이전트 / 생성-검증 쌍 / 팀 — 신호 표로 고정 | 이분법(직접 vs 팀)은 중간 규모 작업을 양극단으로 밀어낸다. 생성-검증 쌍이 구현 작업의 최소 단위 |
| 과판정 방지 | 애매하면 가벼운 쪽 시작 + patterns.md 전환 신호로만 승격, 판정·근거 한 줄 보고 의무 | 관문의 오버헤드가 크면 관문 자체가 우회된다(team-review Phase 0과 동일 철학). 무언의 판정은 우회와 구분 불가 |
| 검증 필수 | 기능 추가·동작 변경 포함 시 reviewer 독립 검증 Phase 생략 불가. 대규모·고위험은 team-review 팀 모드로 승격 | 검증이 선택 항목이면 바쁜 작업일수록 먼저 생략된다. 13절 3항(스펙 대비 리뷰)의 실행 경로를 orchestrate에 내장 |
| 하이브리드 표준 스켈레톤 | 5-Phase 고정 순서, 필요 Phase만 선택, Phase별 패턴 독립 판정, 경계는 `_workspace/` 파일 | 매번 즉석 조합하면 검증 Phase가 빠지거나 경계가 메모리로 넘어가 추적 불가가 된다 — 스켈레톤은 조합의 기본값을 제공하고 생략을 명시적 결정으로 만든다 |
| PR 본문 템플릿(동시 변경) | doc-writer 템플릿 5번 신설, branch-workflow 마무리 4단계가 참조. `Closes #N`으로 work-tracker와 연결 | 문서와 같은 원리 — PR 형식이 매번 다르면 리뷰어가 매번 읽는 법을 새로 배운다. 템플릿 단일 원본은 doc-writer(중복 정의 금지) |
| 스킬 우선순위 (reviewer 검증 F3·F8~F10) | CLAUDE.md 트리거 절에 우선순위 신설 — 내부: 특화 스킬 > orchestrate 게이트, 외부: 하네스 스킬 > 내장·플러그인·sc:*(영어 트리거 포함). 외부 유출 관찰은 harness-review 우회 신호에 추가 | 관문화로 orchestrate 트리거가 넓어지자 내부 특화 스킬과 이중 매칭이 생기고, 외부 스킬(sc:implement, gstack ship·checkpoint)과의 축자 충돌이 라우팅을 오염 — 항상 로드되는 포인터(CLAUDE.md)가 우선순위를 결정하는 것이 "리뷰해줘 유출" 해소 선례와 동일 메커니즘 |
| 규모 임계 단일 원본 (F6) | "변경 파일 6+"의 원본은 orchestrate Phase 0-1 판정 표 — team-review 등은 참조만 | 같은 임계가 3곳 독립 하드코딩되면 한 곳만 바뀌었을 때 같은 작업이 스킬마다 다른 규모 판정을 받는다 |
| 저위험 트리거 겹침 수용 (F11) | doc-writer "report"↔docx(Word 산출물), "검증"↔내장 verify 등은 문구 조정 없이 harness-review 주간 관찰로 흡수 | 겹침의 실해악은 유출 발생 빈도로만 판정 가능 — 관찰 전 선제 조정은 description을 실사용과 어긋나게 만든다 |
