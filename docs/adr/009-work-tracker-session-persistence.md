# ADR 009 — 세션 영속 작업 추적 (work-tracker, ccpm 패턴)

- **날짜**: 2026-07-13
- **변경 내용**: AGENTS.md 14절 신설 + `work-tracker` 스킬 신설. 세션을 넘는 작업의 상태를 **해당 프로젝트 저장소**에 영속화한다 — GitHub 원격이 있으면 GitHub Issues(에픽 이슈 + 태스크 체크리스트 + 진행 로그 코멘트), 없으면 저장소의 `docs/backlog.md`. 완료는 PR `Closes #N`으로 코드와 함께 닫는다.
- **대상**: AGENTS.md 14절(신설), `.agents/skills/work-tracker/`, CLAUDE.md 트리거, README
- **사유**: `_workspace/`는 미보존(ADR 001)이고 세션 컨텍스트도 세션과 함께 소실된다 — 세션 경계에서 "어디까지 했고 다음이 뭔지"가 유실되어 다음 세션이 git 이력으로 역추적해야 했다. ccpm(GitHub Issues 기반 작업 관리, 2026년 인기 하네스)의 패턴을 이 하네스 구조에 맞게 채택한다(2026-07-13 사용자 승인).

## 결정

| 결정 | 내용 | 근거 |
|---|---|---|
| 저장소 = 프로젝트 저장소 | 상태를 하네스·`_workspace/`가 아니라 대상 프로젝트의 GitHub Issues(또는 `docs/backlog.md`)에 둔다 | 상태는 코드와 같은 곳에 있어야 어긋나지 않는다. 하네스 저장소·`_workspace/`는 각각 공용 규칙·미보존 산출물 전용이고, 설치처가 바뀌어도(ADR 005) 프로젝트 저장소는 따라온다 |
| 등록 기준 고정 | 한 세션에 끝나지 않을 작업만 등록 (여러 PR·며칠 규모·사용자 명시 요청) | 전수 등록은 이슈 무덤을 만든다 — 지연 생성(ADR 007)과 같은 철학: 필요가 증명된 것만 만든다 |
| ccpm의 worktree 병렬 실행은 미채택 | 이슈 기반 상태 영속만 가져오고, git worktree 병렬 에이전트 실행은 도입하지 않는다 | 병렬 실행은 orchestrate가 이미 담당한다(중복 도입은 패턴 충돌). 이 하네스의 갭은 실행이 아니라 **세션 간 상태**였다 |
| 기존 스킬과의 연결 | 등록→branch-workflow 시작, 태스크=스펙 완료 기준 단위(13절), 완료=PR `Closes #N`(branch-workflow 마무리) | 새 흐름을 만들지 않고 기존 관문(브랜치·스펙·PR)에 상태 추적을 끼운다 — 별도 흐름은 우회된다 |
| 외부 발행물 취급 | 이슈 본문·코멘트에 시크릿 금지(가드레일 3절 재확인) | GitHub Issues는 저장소 밖 외부 서비스 발행물이다 |
