# ADR 012 — 설치처 프로필(개인/사내)과 하네스 업데이트 대기 큐

- **날짜**: 2026-07-14
- **변경 내용**: ① 설치처 프로필(개인/사내) 신설 — REGISTRY.md 상단에 기록(harness-install 인터뷰). ② **사내 프로필에서는 추적 하네스 파일의 수정·커밋·푸시를 전면 금지**. ③ 사내에서 발견된 하네스 개선 사항은 `_workspace/harness-updates.md`(미추적) **대기 큐**에 기록하고, 개인 설치처에서 metaskill "대기 큐 적용 시나리오"로 반영. ④ 4절 `_workspace/` 미보존 원칙에 큐·점검 마커 예외 명시.
- **대상**: AGENTS.md 4·5절, harness-install(인터뷰·REGISTRY.md 구조·완료 판정), harness-review(무결성 점검·제안 생성 분기), metaskill(프로필 게이트·적용 시나리오), README
- **사유**: 사용자 확정(2026-07-14) — "하네스를 사내에서도 쓰는데 저장소는 내 개인 레포다. 사내에서 개인 git에 푸시하면 안 된다. harness-review 시 레포를 수정할 게 아니라 _workspace 같은 gitignore 영역에 어떻게 업데이트해야 할지 정보를 남겨야 한다."

## 결정

| 결정 | 내용 | 근거 |
|---|---|---|
| 프로필 위치 = REGISTRY.md | 설치처 프로필은 REGISTRY.md 상단에 기록 | REGISTRY.md가 이미 "설치 환경별 미추적 단일 원본"(ADR 005) — 설치처 속성의 자연스러운 집 |
| 사내 = 수정도 금지 (푸시만 금지가 아님) | 사내에서는 추적 파일을 커밋 없이 고치는 것도 하지 않는다 | 로컬 수정·커밋은 결국 푸시가 필요해 발산하고, 미푸시 변경은 개인 머신과 영구 어긋난다 — 수정 자체를 이월하는 것이 유일한 안정 상태 |
| 큐 위치 = `_workspace/harness-updates.md` | 미추적 영역에 대기 항목을 기록. 4절 미보존 원칙에서 예외로 지정(점검 마커와 함께) | 사용자 지정("_workspace 나 다른 gitignore 영역"). 미추적이어야 사내 데이터가 개인 저장소에 섞이지 않는다 |
| 항목은 자립적으로 | 항목 형식(제목/상태/대상 파일/변경 내용/사유·근거)은 기록 세션의 맥락 없이 개인 머신에서 그대로 적용 가능한 수준으로 구체화 | 큐는 git으로 전달되지 않아 사용자가 수동으로 옮긴다 — 옮겨진 텍스트만으로 적용이 가능해야 한다 |
| 적용은 metaskill 단일 경로 | 개인 설치처에서 "하네스 업데이트 적용" → 항목 검토·적용 → 상태 `적용됨` 갱신 | 관찰(harness-review)과 실행(metaskill)의 역할 분리 유지. 상태 갱신이 중복 적용을 막는다 |
| 미기록 프로필 = 확인 후 진행 | 프로필이 없으면 하네스 파일 수정 전 사용자에게 확인해 기록. 원격 PR 세션만 개인 간주 | 안전 기본값 — 사내에서 실수로 수정하는 비용이 개인에서 한 번 묻는 비용보다 훨씬 크다. 원격 세션은 PR 작업 자체가 개인 저장소 흐름이라 모순이 없다 |

## 기각한 대안

- **사내용 포크/미러 저장소**: 사내 정책상 외부 저장소 연결 자체가 문제일 수 있고, 두 저장소의 이력 동기화가 새 유지 비용이 된다.
- **로컬 커밋 허용 + 푸시만 금지**: 위 표 2행 — 발산 상태를 만든다.
- **큐를 추적 파일로**: 사내 관찰 데이터(어떤 작업이 반복됐는지 등)가 개인 저장소에 커밋되는 것 자체가 정책 위반 소지 — 미추적이 정합.
