# 설계 제안: ponytail의 코드 최소주의를 이 하네스에 적용

- **상태**: 제안(pre-ADR) — 사용자 승인 시 metaskill이 구현하고 ADR 017로 기록
- **날짜**: 2026-07-15
- **분석 대상**: https://github.com/DietrichGebert/ponytail (MIT, v4.8.4)
- **적용 대상**: 이 루트 하네스(nohdol-harness)

## 1. 배경

ponytail은 "게으른 시니어 개발자" 철학으로 **AI 에이전트가 최소한의 코드만 쓰도록** 유도하는 하네스다. 이 프로젝트도 AGENTS.md·스킬·훅으로 구성된 에이전트 하네스이고 멀티 CLI(Claude Code·Codex)를 지원하므로 구조가 겹친다. "ponytail을 분석하고 이 프로젝트에 어떻게 적용할지 설계하라"는 요청에 대한 답이다.

## 2. ponytail 분석

### 2.1 철학 — 결정 사다리(Decision Ladder)
코드를 쓰기 전에 7단계를 순서대로 밟는다:
1. 필요성(YAGNI) — 애초에 필요한가
2. 기존 코드 재사용 — 코드베이스에 이미 있는가
3. 표준 라이브러리
4. 네이티브 플랫폼 기능
5. 이미 설치된 의존성
6. 한 줄로 표현 가능한가
7. 그제서야 최소 구현

핵심 규율: **"문제를 이해한 뒤에 사다리를 오른다 — 이해 대신이 아니라."** / **"동작하는 가장 짧은 diff가 이긴다, 단 문제를 이해한 뒤에."**

### 2.2 게으름이 적용되지 **않는** 영역 (전면 엄밀성 유지)
문제 이해(전체 읽기·흐름 추적), 신뢰 경계의 입력 검증, 데이터 손실을 막는 에러 처리, 보안·접근성, 하드웨어 캘리브레이션, **명시적으로 요청된 기능**. → 최소주의가 무모함으로 변질되는 것을 막는 안전판.

### 2.3 문서 요구
사소하지 않은 로직에는 **"실행 가능한 검증 하나를 남긴다 — 실패하는 가장 작은 것"**(프레임워크 없는 assert 데모나 최소 테스트).

### 2.4 전달 메커니즘
- **AGENTS.md 룰셋** — 위 규율의 원본(규칙 운반체)
- **훅(Node.js)** — 모드 트래커(`lite`/`full`/`ultra`/`off`), **Agent 도구로 생성되는 모든 서브에이전트에 룰셋 자동 주입**(정규식 스코프 `PONYTAIL_SUBAGENT_MATCHER`), statusline
- **스킬/슬래시 명령 6종** — `/ponytail`(모드), `/ponytail-review`(diff 과설계 → 삭제 목록), `/ponytail-audit`(저장소 전체 스캔), `/ponytail-debt`(유예 항목 원장), `/ponytail-gain`(벤치마크), `/ponytail-help`
- **Claude Code 플러그인** — `/plugin marketplace add … && /plugin install ponytail@ponytail`

`/ponytail-review`의 판정 태그 5종: `stdlib`(표준 라이브러리 재구현), `native`(플랫폼/의존성 중복), `yagni`(단일 구현 추상화·미사용 설정), `delete`(죽은 코드·투기적 기능), `shrink`(더 짧게). 출력: `L<line>: <tag> <what>. <replacement>.` + `net: -<N> lines possible` / `Lean already. Ship.` — **정확성·보안·성능은 명시적으로 범위 밖**(별도 리뷰 패스).

### 2.5 근거의 무게 (정직한 평가)
벤치마크는 FastAPI+React 한 저장소, 12개 티켓, **Haiku 4.5, n=4** 기준 -54% LOC / -22% 토큰 / -20% 비용 / -27% 시간. 방향성 근거로는 유효하나 **표제 수치를 채택 명분으로 과신하지 않는다** — 채택 가치는 수치가 아니라 **이 하네스에 없는 규율 축**을 채운다는 정합성에 있다.

## 3. 정합성 진단 — 이미 있는 것 vs. 공백

| ponytail 요소 | 이 하네스의 대응 | 판정 |
|---|---|---|
| 게으름 = 최소주의 (하네스 비대 방지) | ADR 007 지연 생성, "미리 만든 스킬은 부채", 8절 신호 ④ 수축·효율, 15절 토큰 효율 | **이미 있음** — 단 **하네스 자산**에 한정 |
| 결정 사다리 (제품 코드 최소화) | 없음 | **공백** ✅ 핵심 |
| 과설계 리뷰 관점 | team-review 관점 축(정확성·보안·테스트·규약·성능·UX) — 단순성 축 **부재** | **부분 공백** ✅ |
| "실행 가능한 검증 하나" | 13절 TDD(실패 테스트 먼저) | 이미 있음(더 강함) |
| 서브에이전트 룰 주입 훅 | 15절 발행 프롬프트 영어 규약 + 에이전트 정의 10절 | 대응 있음, 방식 다름 |
| 유예 원장(`/ponytail-debt`) | work-tracker(이슈)·backlog.md | 이미 있음 |
| 모드/플러그인/statusline | (해당 없음) | 불필요 |

**결론**: 이 하네스의 최소주의는 **하네스 자신**을 향한다(스킬을 함부로 만들지 마라). ponytail이 채우는 공백은 **에이전트가 하위 프로젝트에 쓰는 "제품 코드"의 분량 규율**이다 — 프로세스(SDD/TDD·리뷰·게이트)는 촘촘하지만 "가장 적은 코드로" 게이트는 없다.

## 4. 적용 설계 — ponytail 자신의 사다리를 적용해서

> ponytail의 사다리 2번(재사용)·5번(기존 의존성)을 이 채택에도 적용한다: **새 스킬·훅·플러그인을 만들지 않고 기존 운반체를 재사용**한다. 규칙 운반체는 문서라는 12절 원칙과도 일치한다.

### Layer 1 — AGENTS.md에 원칙 신설 (핵심 payload)
16절 **"코드 최소주의(제품 코드)"** 신설:
- 결정 사다리 7단계를 이식
- "문제 이해 뒤에 오른다" / "동작하는 가장 짧은 diff" 규율
- **적용 범위 = 하위 프로젝트 제품 코드**(implementer 에이전트·코드 작업). 하네스 자산 지연 생성(ADR 007)과 **별개 축**임을 명시 — 하나는 하네스 비대, 하나는 제품 코드 분량.
- **예외 영역 명시**: 2.2의 엄밀성 유지 영역. 특히 3절 가드레일(보안·검증·데이터 손실)·13절 완료 기준은 최소주의로 **완화 불가** — "짧은 코드"가 "빠뜨린 코드"의 알리바이가 되면 안 된다.
- 이유: 규칙의 유일한 운반체는 문서(12절). 훅·플러그인은 cwd 로드 편의 장치라 원본이 될 수 없고 Codex가 못 읽는다.

### Layer 2 — team-review에 관점 축 1개 추가 (새 스킬 아님)
team-review 관점 표에 **"단순성/과설계"** 축 추가:
- 판정 태그 = ponytail 5종(stdlib·native·yagni·delete·shrink)
- 출력 = 삭제 목록 + `net: -N lines` 형태를 리포트에 포함
- **정확성·보안 축과 분리**(ponytail 원칙 그대로) — 이 축은 "무엇을 지울 수 있나"만 본다
- UI diff에만 UX 축을 선발하듯, **비자명 로직 diff에만 이 축 선발**(작은 diff엔 낭비 — team-review 규모 스케일링과 동일)
- 이유: 관점 팬아웃은 이미 있는 메커니즘. `/ponytail-review`를 새로 만들면 ADR 007(지연 생성) 위반이자 CLAUDE.md의 리뷰→team-review 라우팅을 우회. **사다리 2번(재사용)의 실천.**

### Layer 3 — 경계 명시 (혼선 방지)
- **제품 코드 shrink는 team-review/PR 리뷰**가, **하네스 자산 shrink는 harness-review ④**가 담당 — 두 축을 섞지 않는다(harness-review는 하네스만 본다).
- `/ponytail-audit`(저장소 전체 과설계 스캔)의 가치는 team-review "팀 모드"로 대상 범위를 diff 대신 저장소로 지정하면 재현 가능 → 별도 스킬 불필요.

## 5. 채택하지 **않는** 것 (명시적 비목표)

| 미채택 | 이유 |
|---|---|
| Claude Code 플러그인 설치 | 11절: 외부 도구는 전역 설치·보조 도구로만. 플러그인 훅은 cwd 로드라 심링크 규율·Codex 비호환과 충돌 |
| Node.js 훅(mode-tracker·subagent 주입·statusline) | 이 하네스 훅은 Python `_common.py` 기반. Node 훅 체인 추가는 사다리 5번 위반(새 런타임 의존성). 서브에이전트 무차별 주입은 스코프 도구·티어·영어 발행 프롬프트(10·15절)를 오염 |
| 모드(lite/full/ultra) 설정 | 강도 조절은 team-review 규모 스케일링·사용자 지시로 이미 가능. env·config 파일은 설치처별 상태 증식 |
| `/ponytail-debt` 원장 | 유예 작업 추적은 work-tracker(이슈)·backlog.md가 담당 |
| `/ponytail-gain` 벤치마크 스코어보드 | 근거 무게(2.5) 대비 유지 비용 과다 |

## 6. 거버넌스

- **라이선스**: ponytail은 MIT. 결정 사다리 문구를 이식하므로 16절 하단과 ADR 017에 **출처·MIT 크레딧**을 명기한다.
- **변경 절차**: 이는 하네스 변경(8절 진화) → 사용자 승인 후 **metaskill**이 실행, **ADR 017**로 결정 기록(6절 ADR 트리거 — 규칙 신설). 16절 신설·team-review 개정은 같은 커밋에 변경 이력 동봉(5절).
- **설치처 프로필**: 이 저장소를 직접 클론한 원격 PR 세션은 **개인 프로필**로 간주(5절) → 수정·푸시 허용.
- **검증**: 16절 신설 후 team-review에 자기 자신을 걸어 "단순성 축"이 실제로 삭제 목록을 내는지 1회 확인(dogfooding).

## 7. 완료 기준 (승인 시)

1. AGENTS.md 16절 "코드 최소주의(제품 코드)" 신설 — 사다리 7단계·예외 영역·ADR 007과의 축 구분 포함
2. team-review SKILL.md 관점 표에 "단순성/과설계" 축(태그 5종·출력 형식·선발 조건) 추가
3. CLAUDE.md 트리거 절에 반영 불필요(team-review 라우팅 유지) — 확인만
4. ADR 017 작성(배경·결정·근거·MIT 출처), 변경 이력 테이블 갱신(AGENTS.md·team-review)
5. dogfooding 1회 통과

## 부록: 왜 "최소 채택"인가

ponytail의 가치는 **철학(제품 코드 최소주의)**이지 그 배송 기계(플러그인·Node 훅·6스킬)가 아니다. 이 하네스는 규칙 운반체가 문서라는 원칙과 지연 생성 규율을 이미 가졌으므로, **원칙 1개 + 리뷰 관점 1개**만 이식하면 배송 기계 전체를 복제하지 않고 효과를 얻는다. 이 설계 자체가 ponytail의 사다리("이미 있는 걸 재사용하고 최소만 추가하라")를 실천한 결과다.
