# 설계 제안: claude-mem의 세션 메모리 착안을 이 하네스에 적용

- **상태**: 채택됨 (2026-07-16, 최소 채택 = 패턴 3개) — 훅 1개 + AGENTS.md 3·4절 규약, 결정 근거 docs/adr/018
- **날짜**: 2026-07-16
- **분석 대상**: https://github.com/thedotmack/claude-mem (Apache-2.0)
- **적용 대상**: 이 루트 하네스(nohdol-harness)

## 1. 배경

claude-mem은 **세션 간 영속 메모리** 시스템이다 — 세션 중 에이전트 활동을 자동 캡처 → AI로 압축 → 다음 세션에 관련 컨텍스트를 자동 주입한다. 이 하네스도 AGENTS.md·스킬·훅으로 구성된 Claude Code 확장이고, "세션 경계에서 상태가 유실된다"는 문제를 work-tracker(ADR 009)로 이미 다루고 있어 비교 지점이 명확하다. "claude-mem을 분석하고 이 프로젝트에 적용할 수 있는 것을 검토·적용하라"는 요청에 대한 답이다.

## 2. claude-mem 분석

### 2.1 구성
- **5개 생명주기 훅**(SessionStart / UserPromptSubmit / PostToolUse / Stop / SessionEnd) — 관찰(observation) 자동 수집·요약
- **Worker 서비스** — Bun이 관리하는 로컬 HTTP API + 웹 뷰어 UI
- **저장소** — SQLite(FTS5) + Chroma 벡터 DB(하이브리드 시맨틱/키워드 검색)
- **mem-search 스킬 + 4개 MCP 툴** — `search`→`timeline`→`get_observations` 3계층 점진적 공개
- **프라이버시** — `<private>` 태그로 특정 내용 저장 제외

### 2.2 핵심 설계 철학 2가지
1. **자동·무개입 전수 캡처** — 사용자 개입 없이 세션 전체를 관찰·압축·저장.
2. **점진적 공개로 토큰 절감** — 색인만 먼저(50~100토큰) → 필터 후 선택 ID만 전체(500~1000토큰) 인출. "필터 후 인출로 ~10배 토큰 절감."

### 2.3 이 하네스와의 중첩·긴장
- **work-tracker(ADR 009)** 가 같은 문제(세션 간 상태 유실)를 **정반대 철학**으로 이미 해결: claude-mem은 "전수 자동 캡처", work-tracker는 "멀티세션 작업만 등록 — 이슈 무덤 방지"(지연 생성 ADR 007, 코드 최소주의 ADR 017과 같은 계열).
- **훅 인프라**를 이미 운영(SessionStart 2종 + tdd-gate git 훅) — claude-mem식 생명주기 훅 패턴은 익숙한 지형.
- **integrator 게이트·explorer 리포트** — `_workspace`에 리포트를 쌓고 메인 루프가 소비하는 구조가 점진적 공개와 맞물린다.

## 3. 채택 설계 (Layer 1 — 착안만, 인프라 없이)

### ① 세션 경계 진행-기록 리마인더 훅 — 채택
work-tracker 흐름 2("세션 종료 시 진행 로그를 반드시 남긴다")는 실행이 기억에 의존해 강제 장치가 없다. claude-mem의 SessionEnd 캡처 착안을 빌려, **SessionStart 훅이 로컬 git으로 이전 세션의 미커밋 작업을 감지**하면 흐름 2/3을 환기한다.
- SessionEnd/Stop 훅은 모델을 못 깨워 지시를 주입 못 한다(harness-review-reminder와 동일 제약) → SessionStart 로컬 감지로 대체.
- 인프라 0 추가(로컬 git 조회뿐, 네트워크 없음, 읽기 전용, fail-open). 기존 훅 관례 재사용.
- 산출물: `.agents/hooks/worklog-reminder.py` + 회귀 테스트 + 스펙(2026-07-16), `.claude/settings.json` 등록.

### ② 점진적 공개(2단) 리포트 규약 — 채택
claude-mem의 "색인 먼저, 선택만 인출" 패턴을 `_workspace/` 리포트에 적용: 발견 많은 리포트 = 요약 인덱스(ID·severity·요지) + ID 참조 상세 2단. integrator·메인 루프가 인덱스로 필터한 뒤 필요한 상세만 재독.
- 인프라가 아니라 **프롬프트 규약**으로 흡수(AGENTS.md 4절 단일 원본 + explorer 출력). 15절 ④ 수축·효율과 목적 일치.
- 짧은 리포트는 인덱스 생략(16절 최소주의 — 규율이 노이즈가 되면 우회된다).

### ③ `<private>` 제외 마커 — 채택
`_workspace`→이슈 요약 이동 시 `<private>` 내용을 외부 발행물에서 제외. 기존 시크릿 가드레일(3절)의 연장선.
- AGENTS.md 3절 가드레일 + work-tracker 흐름 2에 반영. 소규모 보강.

## 4. 비목표 (기각한 것과 이유)

- **Worker 서비스 + SQLite + Chroma + Bun/uv 런타임 스택 — 기각.** ADR 005(미추적 이식형 하네스)와 정면 충돌. 이 하네스는 의도적으로 마크다운+python 훅만으로 설치처를 따라오게 설계됐고, 상시 로컬 HTTP 워커·벡터 DB·별도 런타임 매니저는 이 무의존·이식성을 깬다. 시맨틱 검색 이득 < 운영 부담.
- **전수 자동 캡처 — 기각.** work-tracker "남발 금지 / 이슈 무덤 방지"(ADR 009)·16절 최소주의와 충돌. 이 하네스의 공백은 '전수 기억'이 아니라 의도적 상태 영속이었고 그건 이미 해결됨. ①은 캡처가 아니라 감지·환기만 한다.
- **mem-search 스킬 + MCP 검색 툴 — 기각.** 벡터 DB 위에서만 의미가 있어 ②의 인프라 기각과 함께 탈락. 과거 작업 조회는 work-tracker 이슈·git 이력으로 충분.
- **claude-mem 코드베이스 복제 — 기각.** 규칙의 유일 운반체는 문서다(12절). ponytail 채택(ADR 017)과 같은 원칙 — 배송 기계 전체가 아니라 착안만 옮긴다.

## 5. 근거의 무게 (정직한 평가)

claude-mem과 이 하네스는 **방향이 다르다**: 전자는 무거운 인프라로 자동 전수 기억, 후자는 가벼운 규약으로 의도적 상태 영속. 그래서 시스템 자체를 이식할 이유는 없다. 채택한 3개는 모두 **인프라 0·기존 철학 정합·저비용**이라는 공통점으로 골랐고, 그중 ②(점진적 공개)가 가장 실질 이득이 크다(재독 토큰 절감은 곱으로 붙는다). ①은 강제 장치가 없던 흐름 2를 보강하되 감지·환기에 그쳐 노이즈 위험을 최소화했고, ③은 기존 가드레일의 명시화에 가깝다.

> 출처: 세션 메모리·점진적 공개·`<private>` 착안은 claude-mem(github.com/thedotmack/claude-mem, Apache-2.0)에서 이식했다. 결정 근거·기각 항목은 docs/adr/018, 훅 스펙은 docs/specs/2026-07-16-worklog-reminder-hook.md.
