## Always-on anchors (Claude-only — details in AGENTS.md)

- **Output language (§15)**: All user-facing output (chat, summaries, reports, questions, PR bodies, documents) is **fixed to Korean**. English is only for §15 model-only internal communication (dispatch prompts, team interim reports, P2P) — never leak English into anything the user reads. When relaying a subagent's English report, do not quote or transliterate it — **digest it and report in Korean**, and immediately before the final report verify that every piece of user-facing text is Korean (§15 guard 1).
- **Routing — skill map (§7)**: Multi-step work (implementation, fixes, refactoring, etc.) + multi-project/team/parallel → `orchestrate` (Phase 0-1 gate — if the verdict is 'perform directly', return immediately). Harness creation/improvement/new project setup → `metaskill`. Review/inspection/verification ("리뷰해줘", "코드 리뷰", "PR 검토") → `team-review` (not the built-in review). Documents/specs/reports/runbooks → `doc-writer`. New install site / missing REGISTRY.md → `harness-install`. Registering/resuming session-spanning work ("작업 등록", "이어서 하자", "하던 작업 뭐였지") → `work-tracker`. Deploy/release → `release`. PR creation → `branch-workflow` + doc-writer PR template. Whole-project status ("프로젝트 상태", "전체 현황", "상태 요약") → `project-status`. Reading/summarizing URL content ("링크 읽어줘", article/document URLs) → `defuddle` (.md URLs and auth-gated pages → WebFetch). Library/framework/SDK usage & docs lookup ("어떻게 쓰지", configuration, migration, library-specific errors) → `context7` (Anthropic/Claude API → `claude-api` (Claude Code built-in); when not installed or for a specific URL → WebFetch/defuddle). Measured usage audit of external tools/plugins ("플러그인 쓸모 있나", "사용 실측", "도구 정리") → `tool-audit` (periodic checks → harness-review; removal execution → metaskill). Unattended autonomous loop ("자율 루프", "무인 실행", "밤새 돌려줘") → `autoloop` (periodic repetition → built-in /loop; manual handoff → carryover). **Continuation ('진행해줘', '계속') also re-passes the orchestrate gate when it involves new implementation, behavior change, or k8s manifest change** — k8s manifest changes go through `infra-specialist` even as a continuation, with namespace resource-limit pre-check (details: AGENTS.md §7). **The transition point where a conversation that began as read-only diagnosis (log checks, kubectl, root-cause investigation) naturally shifts into code modification is subject to the same re-judgment — even without an explicit '구현해줘' from the user, the moment the first Edit/Write touches product code is the gate trigger** (the gate-reminder hook mechanically reminds once per session — real case & rationale: ADR 028). **Concurrent handling of independent tracks (e.g. infra change + code change, different repos) must actually re-invoke orchestrate and follow the simultaneous-dispatch convention** — self-narrated judgment misses the simultaneous-dispatch convention (real case: changelog 07-21). **Subagent dispatch defaults to `run_in_background` regardless of whether parallel work exists** — a "no parallel work" justification is itself a bypass signal (real case: changelog 07-21). **This holds even when the next action depends on the agent's result** (user decision 2026-07-24): dispatch in background, tell the user what is running, end the turn, and continue the dependent step from the completion notification — the session stays free for the user to keep talking. Foreground is reserved for short lookups (≲1 min) whose result the same response must inline. **The default owner of sub-project product-code implementation is implementer** (3+ files — high-risk work (infra/security, §3) gets team judgment regardless of file count. Main-loop direct implementation is an official path only for ≤2 files, low risk, plus an independent reviewer — orchestrate 0-1 implementation-owner rule). **Implementation taken over via work-tracker "이어서" / carryover resume also re-passes the gate** — going straight in right after resume is a bypass signal (real case: changelog 07-21). **Collection/summary-type subagent dispatch uses the `explorer` type, not general-purpose** (root-cause investigation → troubleshooter; verification → reviewer — §7 item 6).
- **Skill priority (§7)**: ① If the request matches a specialized skill's trigger, **the specialized skill takes precedence over the orchestrate gate** (review → team-review, documents → doc-writer, harness → metaskill). ② For the same trigger, **harness skills take precedence over built-in, plugin, SuperClaude, and gstack skills** — "implement/fix/build" → `orchestrate`; "debug/investigate/QA·test-and-fix" → troubleshooter after the orchestrate gate (fixes via the implementer loop); "design" → architect after orchestrate; "review/code review/security audit" → `team-review`; "deploy/land/ship to production" → `release`; "create a PR/ship/push to main" → `branch-workflow`; "resume/이어서/checkpoint/where was I" → `work-tracker`. **Merging is always the user's call** — never delegate to auto-merging external skills (land-and-deploy etc.). External skills are used only as auxiliary tools inside harness procedures (routing that leaks to external skills bypasses the harness's judgment, templates, and guardrails wholesale).

- **clarify-first (§13 item 0)**: For design/architecture-level work (module boundaries, technology choices, refactoring direction, etc.), before starting: ⓐ enumerate the prompt's implicit assumptions, missing constraints, and decision criteria; ⓑ ask back only the questions whose answers change the approach (AskUserQuestion); ⓒ declare the rest as explicit assumptions and proceed — do not fix an approach first and append questions afterward (forking questions come first). **No question spamming**: things discoverable by lookup, obvious defaults, and answers that leave the approach unchanged are not asked — declare assumptions and proceed (compatible with the base 'act when sufficient'). Trivial single fixes are excluded.
